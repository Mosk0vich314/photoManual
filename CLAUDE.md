# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A mobile-first React app that turns `Photo_Manual.pdf` (a 77-page Italian photography &
color-grading manual) into an interactive reference. The app's UI text and all content are
**in Italian** — keep it that way. On top of the extracted PDF it also carries hand-authored
additions: a "Profili Film" section (6 Lightroom film profiles) and extra manual chapters not
in the PDF (e.g. the RGB-curves lesson with its interactive curve playground).

## Commands

```bash
npm install
npm run dev        # Vite dev server at http://localhost:5173
npm run build      # static build to dist/  (vite base is './', so it runs from any path)
npm run preview    # serve the production build
npm run extract    # python extract.py -> regenerates src/data/manual.json from the PDF

pip install -r requirements.txt   # PyMuPDF, needed only for extract.py
node shoot.mjs [route] [name]     # mobile screenshot via puppeteer-core + installed Chrome
```

There is no test runner or linter configured.

**Deploy:** pushing to `main` triggers `.github/workflows/deploy.yml`, which builds and
publishes to GitHub Pages (live at `https://mosk0vich314.github.io/photoManual/`). There is no
manual deploy step — just push. Confirm a run via the public Actions API if `gh` isn't installed.

**`shoot.mjs` gotcha:** the PWA service worker caches the app shell, so a fresh `page.goto` to a
`#/route` can render the cached **Home** instead of the target route (this silently wasted time —
every route looked like Home). Before screenshotting a non-home route, first load `/`, then in the
page unregister all `navigator.serviceWorker` registrations and clear `caches`, *then* `goto` the
hash URL. Puppeteer `page.mouse.down/up` does dispatch pointer events, so the curve playground's
pointer handlers are testable headless.

## The data pipeline is the heart of the project

Most content is **extracted from the PDF**, not authored by hand — so most "content bugs" in the
24 looks and the PDF theory chapters are fixed in `extract.py`, not in JSX. Four data sources feed
the app, all merged by `src/data/index.js`:

1. **`src/data/manual.json`** — generated by `extract.py`. Do not edit by hand; re-run
   `npm run extract`. **Re-extraction overwrites this file only** — the three hand-authored
   sources below survive, by design.
2. **`src/data/styleMeta.json`** — hand-curated per-style metadata (harmony, chromatic axes,
   color swatches, filter tags, and the scene→look table for the picker). Edit this directly.
3. **`src/data/extraChapters.json`** — hand-authored manual chapters that are **not in the PDF**
   (e.g. *Le Curve di Colore (RGB)*). Same block model as `manual.json`; each chapter has an
   `insertAfter: <slug>` and index.js splices it into `CHAPTERS` right after that slug. This is
   how you add a brand-new chapter without touching the PDF pipeline.
4. **`src/data/filmProfiles.json`** — the "Profili Film" section (6 Lightroom film profiles,
   separate from the 24 looks). Drives `/profili`. Its `.xmp` files live in `public/profiles/`.

`index.js` splices in `extraChapters`, attaches `styleMeta.styles[num]` onto each style chapter as
`chapter.m`, splits chapters into `THEORY` / `STYLES` / `APPENDICES`, exposes `PROFILES`, and
provides lookup helpers (`styleBySlug`, `chapterBySlug`, `profileBySlug`) plus full-text `search()`.

### How extract.py works (read before changing extraction)

PyMuPDF (`fitz`), not `pypdf` — pypdf drops spaces at font boundaries. The parser is driven
entirely by the LaTeX font names in the PDF, which encode structure reliably:

| Font (size)        | Meaning                              | Becomes |
|--------------------|--------------------------------------|---------|
| `SFBX1728` (17pt)  | chapter / style / appendix title     | new chapter |
| `SFBX1440` (14pt)  | section heading                      | `heading` block |
| `SFBX1095` "SCHEDA RAPIDA" | quick-card marker             | `scheda` block |
| `SFBX1200` (bold)  | inline parameter label               | run `k:"label"` |
| `SFTT*` (mono)     | slider value (e.g. `-15 / -25`)      | run `k:"value"` |
| `SFTI*` (italic)   | "Esempio visivo" lead-in / captions  | `example` / `caption` block |
| `SFRM*`            | body text                            | run `k:"text"` |
| size ≤ 8.5         | diagram tick labels / wheel degrees  | dropped as noise |

Other key behaviors: canonical titles come from the PDF's TOC (`get_toc()`), not the body
(the body titles are ALL CAPS); end-of-line soft hyphenation is rejoined; Scheda Rapida rows
are split by x-position (label column `x0≈71`, value column `x0≈200`).

### Output data model (`manual.json`)

```
{ meta, chapters: [ {
    id, slug, kind: "theory"|"style"|"appendix", num, title, film, director, page,
    blocks: [
      { type:"heading", text }
    | { type:"para", runs:[ {t, k:"text"|"label"|"value"|"emph"} ] }
    | { type:"example", text }      // "Esempio visivo: …"
    | { type:"caption", text }      // diagram caption
    | { type:"scheda", rows:[ {label, value} ] }
    ] } ] }
```

## App structure

- Routing: `HashRouter` (works on static hosting / `file://`). Routes live in `src/App.jsx`,
  which also renders the sticky topbar and the 5-tab bottom nav.
- Pages (`src/pages/`): `Home`, `Looks` (gallery + filters), `StyleDetail`, `Profili` (film-profile
  gallery), `ProfileDetail`, `Manuale` (TOC), `ChapterReader`, `Picker` (scene→look), `Search`.
- "Profili Film": gallery at `/profili`, detail at `/profili/:slug`. Reached from a full-width tile
  on Home (it is **not** a bottom-nav tab — the nav stays 5 tabs). Each detail page offers an `.xmp`
  download served from `public/profiles/<slug>.xmp`; those files are precached for offline use
  (`vite.config.js` workbox `globPatterns` includes `xmp`, with a raised file-size cap).
- `src/components/Blocks.jsx` renders a chapter's block array. Its `widgets` prop injects an
  interactive component **after a heading with a given text**; the chapter→widget map lives in
  `ChapterReader.jsx` (`CHAPTER_WIDGETS`, keyed by chapter slug). Examples: `HueWheel` into
  *Teoria del Colore* (`"La ruota e i gradi"`), and `CurvePlayground` into *Le Curve di Colore*
  (`"Il banco di prova"`). Use the same mechanism for new interactive widgets.
- `CurvePlayground.jsx` is a self-contained draggable curve editor (RGB/R/G/B channels →
  monotone-cubic LUTs applied per-pixel to a procedurally generated, copyright-free test scene on a
  `<canvas>`). Pointer-driven (touch-friendly): single svg-level handler grabs the nearest point,
  adds one only when the tap is on the curve, and removes via the explicit "Rimuovi punto" button.
- `SchedaRapida.jsx` re-highlights numeric tokens inside the quick-card values.

## Conventions

- Two type families: a serif (`--font` default, the manual's voice) and a UI sans applied via
  the `.ui` class (nav, chips, labels). Colors are CSS variables in `src/styles.css`
  (`--gold` is the accent, `--value` is the slider-value badge color). Dark cinematic theme.
- Some styles have no parenthetical film in the TOC (e.g. Stile 2 "Bergman") → `film` is
  `null`. Components must fall back (usually to `m.harmony`); don't assume `film` exists.
- Style slugs are `stile-<num>-<title>`; link with `/look/<slug>`. Other chapters:
  `/capitolo/<slug>`. Film profiles use a plain slug (e.g. `kodak-portra-400`); link with
  `/profili/<slug>`, and the matching download is `public/profiles/<slug>.xmp`. Helpers in
  `data/index.js`.
